# [Empirical Evaluation of Reasoning LLMs in Machinery Functional Safety Risk Assessment and the Limits of Anthropomorphized Reasoning](https://www.mdpi.com/2079-9292/14/18/3624)

## 摘要

> 建立用於功能安全工作流程（如 PLr 判定）中評估 LLM 的嚴謹基準，支持「LLM 的推理更偏向表面續寫，而非真正推論」的批評。決定性的安全關鍵分類任務需要嚴格的規則約束提示與審慎的檢索治理，而非依賴對模型推理能力的假設

## 盲區

- PLr (Required Performance Level)：一種基於分級的風險估算方法，其參數 S、F 與 P 分別對應傷害嚴重度、操作員暴露於危險區域的時間或頻率，以及避免危害的可能性
    - 傷害嚴重度（S）分為 S1（輕微）與 S2（嚴重）。估算傷害等級
    - 暴露頻率（F）或暴露於危險的時間被分類為 F1（偶爾或短時間）與 F2（頻繁或長時間）。衡量在危險區域中停留的時間
    - 避免危害的可能性（P）分為 P1 與 P2，可避免事故則選擇 P1；若幾乎不可能避免，則選擇 P2。其判定依據危害是否可被識別或避免
    - 上述參數的組合會使風險由低至高增加，組成圖2風險圖
- ISO 12100：規定在機械設計中達成安全之基本術語、原則與方法
    - 說明風險評估與風險降低之原則
    - 規定在機器生命週期的相關階段中進行危害識別以及風險估算與評估之程序

## 研究動機與背景

- 關鍵研究缺口：
    - 缺乏對推理模型在確定性、安全關鍵 PLr 分類中的實證評估
    - PLr 估算中推理偏差與幻覺影響尚未被充分探討
    - 在功能安全風險評估中，缺乏針對推理模型的結構化基準測試方法與實證評估
- 轉換為：
    - RQ1：推理模型是否能在功能安全標準約束下，可靠地執行結構化風險分類任務，例如 PLr 估算？
    - RQ2：推理模型在機械風險評估領域之確定性分類問題中呈現何種限制？
    - RQ3：結構化提示的依賴如何影響功能安全情境中推理模型輸出的一致性與有效性？
- 安全風險評估
    - 聚焦於由 ISO 12100 與 ISO 13849 標準所規範之機械領域
    - LLM 在結構化危害分析中的潛力與限制，並指出其在功能安全等確定性任務中的效果高度依賴結構化提示、領域適配與專家監督
    - 在風險分析任務中，有效與 LLM 互動往往依賴結構化、基於規則的提示。若缺乏領域特定指引，LLM 可能產生不一致或誤導性輸出
    - 強調 LLM 雖具備生成初步危害識別與情境建構的潛力，但在確定性、規則約束的風險評估中仍存在不足
- LLM 在推理任務中的能力與限制
    - 人們樂觀地認為，在正確引導下，LLM 可能接近一般推理能力

    > 然而，多項近期研究對 CoT 驅動推理的假設提出批判性檢視
    
    - LLM 雖能生成局部有效的推理步驟，但在全局證明策略上失敗，顯示其傾向於貪婪式啟發推理而非系統性探索
    - 即使是邏輯上無效或無關的 CoT 軌跡，也可能提升模型在複雜任務上的表現，意味著成功可能源於對 token 模式的熟悉，而非邏輯正確性
    - 顯示 CoT 的收益往往來自表層 token 動態，而非真正的演繹推理能力，對 LLM 推理能力的過度高估提出質疑
    - 將中間文本標註為「思考」會危險地擬人化 LLM，掩蓋 CoT 可能僅透過表層 token 模式改善輸出
    - 使用 CoT 提示的 LLM 存在「推理斷崖（reasoning cliff）」現象：在簡單任務中表現良好，但隨著任務複雜度上升，準確率會突然崩潰
    
    > 研究指出 CoT 輸出未必能透明反映模型決策過程，因此不適合用於可靠審計，並削弱其在高風險應用中作為可解釋性保障的假設

    - 近期研究對 LLM 推理能力呈現混合觀點：雖然 CoT 與自一致性方法可提升基準分數，但許多研究指出其成功可能來自 token 模式匹配，而非真正推理能力，仍存在「能力湧現」與「工程化幻覺」之間的爭論
    - 減輕幻覺與事實錯誤的一項關鍵策略是檢索增強生成（Retrieval-Augmented Generation, RAG），其透過外部知識來源為模型輸出提供依據

    > 提示工程能提升可靠性與可解釋性，但無法保證事實基礎或符合法規要求。這些限制凸顯了結構化提示與 RAG 在確保符合正式安全標準方面的重要性

    - 缺乏結構化基準與評估協議時，很難判斷 LLM 在安全關鍵任務中的可信度

## 研究

- 評估六個最先進模型（Claude-opus、o3-mini、o4-mini、GPT-5-mini、Gemini-2.5-flash、DeepSeek-Reasoner），並涵蓋六種提示策略與源自 ISO 12100 與 ISO 13849 標準之基準資料集兩種(共享相同 schema 與標註（gold labels），但在語言表達上不同)：標準 ISO 風格的危害描述（變體1）以及工程師撰寫的自由文本情境（變體2）
- 以 LangGraph [34] 作為核心編排引擎
- 模組化提示策略（基於規則、RAG 與混合式）透過 LangChain chains [35] 實作
- ChromaDB [36] 向量資料庫
- 主要來源為開源資料集 [7,37]
- 檢索之上下文透過提示佔位符注入，以評估類比推理能力
- 評估資料集
    - 變體1——ISO 標準化情境（N = 100）
        - 此變體將評估限制在與 ISO 12100 附錄 B 高度一致的語境中。每個案例遵循固定結構（危害類型、來源、潛在後果、任務、環境），並明確提供風險參數：嚴重度（S）、暴露頻率（F）、避免可能性（P），以及參考 PLr。由於文本由標準化欄位自動生成且符合 ISO 分類體系，此變體可視為合成但受控的資料集，用於提供明確術語下的標準基準。
    - 變體2——功能安全工程師撰寫情境（N = 100）
        - 此變體用於測試對真實工程語言的泛化能力。情境由功能安全工程師基於工業實務撰寫，並與變體1使用相同 schema 與標註，但自由文本刻意避免直接使用 S/F/P 的 ISO 標籤。相反，風險因素以操作性敘述隱含表達，例如「在傳動機構附近頻繁短暫停機」、「受限間隙且停止反應延遲」或「在設定期間手部進入夾點區域」。此設計模擬工業安全評估中常見的危害描述方式。
- 六種不同提示策略
    1. Zero-shot Prompt [ZERO_SHOT]：僅提供原始危害描述，不附加額外規則或指引
    2. **顯式 ISO 規則整合 [WITH_RULES]：在 zero-shot 基礎上加入 ISO 13849-1 附錄 A 規則**
    3. Chain-of-Thought（CoT）[PURE_CoT]：提供逐步推理指令，但不明確提供 ISO 規則
    4. CoT + 規則 [COT_WITH_RULES]：結合 CoT 推理步驟與 ISO 規則
    5. CoT + 規則 + RAG [WITH_RULES_RAG]：在前述基礎上加入 RAG 檢索得到的歷史危害範例
    6. 規則 + RAG 範例（無 CoT）[COT_WITH_RULES_RAG]：類似前者，但不包含 CoT 指令
- 提示佔位符
    - {description}：危害情境之自然語言描述，用於測試模型對情境理解與風險映射能力。
    - {iso_rules_info}：注入 ISO 13849-1 附錄 A 之規則與風險圖映射邏輯，使任務由開放推論轉為規則約束推理。
    - {COT_step_by_step_instruction}：提供逐步分析指引（嚴重度、頻率、避免可能性），以強制結構化推理流程並提升可解釋性。
    - {rag_examples}：由檢索模組提供之歷史危害範例，包含 S/F/P 與 PLr 標註。
- 三大供應商的六個生產模型，以及一個專門的推理模型，涵蓋三個範疇：
    1. 專用推理架構 OpenAI o 系列：
        - o3-mini
        - o4-mini
        - DeepSeek Reasoner
    2. 成本／延遲最佳化的「mini／flash」變體：
        - Google Gemini 2.5 Flash
        - OpenAI GPT-5 mini
    3. 作為上限基準的高階通用模型
        - Anthropic Claude Opus 4.1
    - PLr（ISO 13849-1）分類任務是一種確定性的多類別分類問題。為消除採樣變異並確保可重現性，所有支援的模型皆採用決定性解碼設定
        - 參數設為：
            - temperature = 0.0
            - top_p = 1.0
            - top_k = 0（貪婪解碼）
        - 不支援者依供應商預設
                            
### 結論

- 以規則為基礎的提示（rule-grounded prompting）可一致地穩定模型表現，在變體1中達到接近滿分的準確率，並在變體2的詞彙變異下恢復可靠性

>相對地，未受限制的思維鏈（Chain-of-Thought, CoT）推理以及結合檢索增強生成（Retrieval-Augmented Generation, RAG）的 CoT 方法會引入波動性、過度預測偏差，以及模型依賴性的性能退化

- 只有規則導向提示能可靠捕捉罕見但高風險的危害。延遲分析顯示，**僅規則提示不僅準確度最高，也最具效率**，而 **CoT 策略則帶來 2–10 倍的計算開銷**
- 決定性的安全關鍵分類任務需要嚴格的規則約束提示與審慎的檢索治理，而非依賴對模型推理能力的假設
- 基於規則的提示方法（𝚆𝙸𝚃𝙷_𝚁𝚄𝙻𝙴𝚂、𝚆𝙸𝚃𝙷_𝚁𝚄𝙻𝙴𝚂_𝚁𝙰𝙶）在整體表現上持續優於零樣本（zero-shot）與未受約束的鏈式思考（CoT）（對應 RQ1）。變體一（ISO 標準化描述）達到接近上限的準確率，而變體二（工程師自由文本）則需要明確規則才能在語意變異下恢復可靠性。
- 模型規模具有關鍵影響
- 自由形式的 CoT 推理引入了顯著不穩定性，使延遲增加約 2–10 倍，並在部分情況下放大了檢索不一致性
- 推理軌跡（S/F/P 流程）經常偏離 ISO 一致的決策邏輯，顯示 CoT 更像是 token 延續機制，而非真正的推理過程。這凸顯了「擬人化風險」，即將具推理能力的 LLM 輸出誤認為人類式推理。

> 只有在輸出被明確 ISO 13849-1 規則約束時，才能實現決定性與正確性

- RAG 在部分模型（如 Claude、o4、GPT-5）中作為提升穩健性與降低延遲的條件性加速器，但在其他模型（如 DeepSeek）中若未經結構化過濾則可能導致錯誤（RQ3）
- 綜合觀察
    - Rule-based prompting 是核心必要條件
    - **CoT 會增加不穩定性與成本**
    - RAG 對不同模型影響不一致
    - PLr E 在 rule-based 下幾乎完全可靠
- 主要結論：
    - **規則提示 > 生成式推理（CoT）**
    - RAG 有條件有效，並非普遍改善
    - **模型規模影響穩定性（o3-mini 較脆弱）**
    - 自由文本會顯著降低非規則模型表現
    - 安全關鍵類別（PLr E）只有 rule-based 才可靠
- **LLM 生成的推理可能營造出錯誤的可靠感**，使表面合理的輸出被誤認為正確推論，進而在安全關鍵場景中導致過度信任。因此，工業部署應強調：
    1. **嚴格基於規則的提示設計，並結合獨立人工驗證**
    2. 提示與模型的適配選擇，以及檢索流程治理，以降低系統性錯誤
